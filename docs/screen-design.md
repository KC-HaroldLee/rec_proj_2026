# 화면 설계 가이드

화면 구성 및 디자인 톤 정리. 초기엔 `login-wireframe.html`, `dashboard-wireframe.html`
(정적 HTML 목업, v0.1)로 시작했으나 실제 구현 완료 후 삭제 — 디자인 토큰은
`app/static/css/style.css`, 실제 화면은 `app/templates/`에 있다.

## 디자인 원칙

세월호참사·가습기살균제참사라는 진지한 주제를 다루는 시민 모니터링 도구이므로,
화려한 장식이나 트렌디한 그라데이션 대신 **차분하고 신뢰감 있는 톤**을 유지한다.

## 타이포그래피

- **본문**: Pretendard (한글 가독성 좋은 산세리프)
- **제목/헤드라인**: Noto Serif KR (진지함/신뢰감을 위해 세리프체 사용)
- 헤드라인은 굵게(700), 본문은 400~500 weight 사용

## 화면 1: 로그인

**목적**: 팀원 20명만 접근 가능한 비공개 도구임을 명확히 전달, 진입 장벽 없이 빠른 로그인

**구성 요소**:
- 상단 eyebrow 텍스트: "사회적참사 권고 이행 모니터링" (프로젝트 정체성)
- 헤드라인: "기록은 흐려지지 않습니다. 다시 확인할 시간입니다." (카피, 변경 가능)
- 부제: 프로젝트 취지 한 줄 설명
- 로그인 폼: 이름 + 비밀번호 (AUTH 테이블 기준, `auth_name` + `password_hash` 대조)
- 하단: "계정이 없으신가요? 팀 운영자에게 문의" + "팀원 20명만 접근 가능" 안내

## 화면 2: 대시보드 (로그인 직후 첫 화면)

**목적**: 로그인 직후 가장 중요한 정보(정체 항목, 전체 현황)를 한눈에 파악, RAG 질의 진입점 제공

**구성 요소 (위→아래 순서)**:

1. **상단 헤더**: 로고/브랜드명, 네비게이션(대시보드/권고 목록/기관별 현황/용어사전), 로그인한 사용자 표시

2. **검색창 (RAG 질의 인터페이스)**
   - placeholder 예시: `몇 년째 "해당사항 없음"으로 남아있는 항목이 뭐야?`
   - 자연어 질문 → 백엔드에서 파라미터 추출 → SQL 집계 또는 벡터검색 라우팅 (ollama-guide.md의 text-to-SQL 패턴 참고)

3. **요약 통계 카드 (4개)**
   - 전체 권고 수 (분야별 세부: 세월호 32 / 가습기 26 / 재난일반 22)
   - N년 이상 이행현황 변화 없음 (정체 항목 수, warn 스타일로 강조)
   - 권고 대상기관 수
   - 최근 30일 시민 피드백 수

4. **정체된 권고 항목 테이블**
   - 필터 칩: 전체 / 세월호참사 / 가습기살균제 / 재난 일반
   - 컬럼: 번호(rec_no) / 권고 내용(+대상기관 태그) / 연도별 변화(점으로 시각화) / 상태 배지
   - 연도별 변화 시각화: 각 연도를 작은 사각형으로 표시하고, 전년도와 `status`가
     동일한 경우 시그널 컬러로 강조 (`same` 클래스) — data-parsing-rules.md의
     복붙 탐지 로직과 연결됨
   - 상태 배지 3종:
     - `stuck` (시그널 배경/텍스트): "N년째 동일"
     - `progress` (완료 컬러): "논의 진행중"
     - `watch` (보조 배경): "이행완료 처리" (단, 실제 이행완료인지는 시민 검증 필요할 수 있음)

5. **최근 시민 피드백 피드**
   - 아바타(이니셜) + 작성자명 + 대상 권고 + 시간 + 피드백 내용
   - FEEDBACK 테이블 데이터를 최신순으로 표시

## 화면 3: 권고 상세 페이지 (`app/templates/recs/detail.html`)

**목적**: 권고 하나를 중심으로 원문·필요성·기관별 연도별 이행현황·시민 검증 기록을 한 화면에서 확인.

**구성 요소**:
- 권고 전문 + 필요성: `REC.content`/`necessity`를 마크다운 렌더링(`app/markdown_render.py`).
  각주는 블록인용(`>`)으로 표현되므로 그대로 `<blockquote>`로 렌더링됨.
- 기관별 이행현황 타임라인: REC_INST(권고+기관) 단위로 나눠서, 각 기관 아래 연도별
  이행현황(status)/향후계획(plan)을 세로 타임라인으로 표시. 대시보드의 정체 항목
  테이블과 마찬가지로 링크(기관) 단위 — 기관별로 텍스트가 다를 수 있다는 전제.
- 연도별 항목마다 시민 피드백 목록 + 작성 폼 (`POST /impl/{impl_id}/feedback`,
  제출 후 `#impl-{id}` 앵커로 돌아옴)
- 관련조사자료(EVID) 목록

## 화면 4: 기관별 현황 (`app/templates/institutions/`)

목록 페이지는 기관별 "정체 아님 비율"(%)만, 상세 페이지는 권고별 행마다 정체/변화
배지 + 최신 연도 status 원문을 나란히 보여준다. "이행률"이 아니라 "정체 아님 비율"인
이유는 design-decisions.md "기관별 '이행률'을..." 항목 참고 — 완료 여부는 시스템이
판단하지 않고 원문을 사람이 직접 읽도록 유도하는 게 의도적인 설계.

## 용어사전 수정/삭제 (`app/templates/partials/term_row.html`, `term_row_edit.html`)

행마다 "수정"/"삭제" 버튼. htmx로 행 단위만 교체(`hx-target="#term-{id}"`) — 수정은
GET `/terms/{id}/edit`로 입력폼 행을 받아왔다가 POST `/terms/{id}`로 저장, 삭제는
POST `/terms/{id}/delete`로 `is_deleted`만 TRUE로 바꾸는 소프트 삭제(schema.md 설계
그대로). 팀원 20명 모두가 서로 수정 가능한 공유 사전이라는 전제라 작성자 제한은 두지
않음 — updated_by만 남겨서 누가 마지막으로 고쳤는지는 추적.

## 화면 5: 검색 결과 페이지 (`GET /search?q=...`, `app/templates/search/results.html`)

대시보드 검색창은 더 이상 htmx로 결과를 그 자리에 끼워넣지 않고, `GET /search?q=...`로
이동하는 일반 폼 제출로 바뀌었다 — 검색 결과가 자기 URL을 가지고(북마크/공유/뒤로가기
가능), 새로고침해도 사라지지 않아야 한다고 판단해서. 페이지 자체에도 같은 검색창이
있어 재검색 가능.

구성: 질문 + 답변(+ 접이식으로 실제 실행된 SQL) + "근거 레코드" 표. 근거 표는
core/rag가 실행한 SQL 결과 원본을 그대로 보여주며, 컬럼명이 `rec_id`/`inst_id`면
자동으로 `/recs/{id}` `/institutions/{id}` 링크로 바뀐다(`app/search_render.py`).
LLM이 만드는 SQL마다 SELECT 목록이 달라서 완벽하지는 않다 — core/rag/qa.py의
프롬프트에 "특정 권고·기관 질문이면 rec_id/inst_id를 같이 SELECT해달라"는 지시를
넣어 보정하는 정도.

## 원문 보기 기능 (설계만 되어 있음, 화면 미제작)

`REC.source`, `IMPL.source` 필드에 `파일명#page=번호` 형식으로 저장해두면:
- PDF: `<iframe src="파일경로#page=40">` 형태로 해당 페이지 바로 열기 가능
- 이미지(캡처본): `<img src="...">` 로 바로 표시

파일 저장 위치(로컬 폴더 vs 파일서버)는 배포 단계에서 결정 필요 (features-backlog.md 참고).
