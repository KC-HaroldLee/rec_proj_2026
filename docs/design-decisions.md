# 설계 배경 / 의사결정 기록

이 문서는 "왜 이렇게 설계했는지" 그리고 "왜 이렇게 안 하기로 했는지"를 기록한다.
스키마만 보면 알 수 없는 맥락을 담아, 나중에 같은 고민을 반복하지 않기 위함.

## 프로젝트 목적

사참위(사회적참사특별조사위원회)가 2022년 발간한 권고사항이, 이후 관련 기관들에 의해
실제로 이행되고 있는지 시민이 직접 추적·검증하는 도구. 법 제48조는 국가기관등이
매년 이행내역을 국회에 보고하도록 규정하는데, 이 연례보고서(2023, 2024...)가
실제로 진전이 있는지, 아니면 같은 문구를 반복 재사용하는지를 감시하는 것이 핵심 가치.

## 순수 RAG가 아니라 구조화 DB + 벡터검색 하이브리드로 간 이유

초기에는 순수 벡터 RAG(문서 → 임베딩 → 유사도검색 → LLM 답변)로 시작했으나,
"몇 년째 해당사항 없음으로 남아있는 항목이 뭐야?" 같은 질문이 실제 핵심 질문으로
드러나면서 방향을 바꿨다. 이런 질문은 본질적으로 "집계"이지 "검색"이 아니다.
벡터검색은 top-k만 가져오므로 정확한 카운팅/그룹핑에 부적합하고, LLM이 안 본 데이터에
대해 추측성 답변(환각)을 만들 위험이 있다.

**결론**: 구조화 DB(PostgreSQL)가 정확한 집계/조회를 담당하고, 벡터검색은
의미 기반 탐색("이 권고와 비슷한 다른 권고는?", "표현만 바뀐 재탕 탐지")을 담당하는
역할 분리 구조로 확정. LLM은 사용자 질문을 파라미터로 변환하는 "질문 해석기" 역할.
(당시엔 벡터검색 = Chroma를 막연히 가정했으나, 실제 구현 시점엔 pgvector로 대체함 —
"벡터 저장소: Chroma 대신 Postgres pgvector 확장" 항목 참고.)

## 기관명을 정규화하지 않기로 한 이유

정부조직 개편(예: 2025년 이재명 정부 조직개편으로 기획재정부 → 재정경제부+기획예산처
분리)으로 기관명이 자주 바뀐다. 처음엔 "기관 이력 테이블"로 개명 이력을 정확히
모델링하려 했으나:

- 1:1 개명(환경부 → 기후에너지환경부)도 있지만, 1:N 분리(기획재정부 → 재정경제부 +
  기획예산처)도 있어 관계가 단순하지 않음
- "이게 같은 기관의 후신인지"조차 실무적으로 논쟁적일 수 있음
- 지금 규모(2000개 레코드)에서 완벽한 자동 매핑 구조는 과설계 위험

**결론**: `INST.name`은 원문 그대로 저장하고, `INST.note`에 사람이 수동으로
승계 관계를 메모하는 정도로 단순화. 조직 위계도 모델링하지 않음 — "권고 대상이었다는
사실"이 중요하지, 정부 조직도 자체가 이 프로젝트의 관심사가 아니기 때문.

## REC_INST, REC_EVID를 다대다로 설계한 이유

실제 원본 문서(사참위 권고집)를 확인한 결과:
- 권고 하나에 대상기관이 9개까지 붙는 사례 확인 (관리번호: 국정원장 등 9개 기관 대상 권고)
- 조사자료도 한 권고에 여러 개(관리번호 19번, 2개), 같은 자료가 여러 권고에서
  공유되는 사례(관리번호 5, 6번) 확인

따라서 REC에 기관/자료를 직접 컬럼으로 넣거나 반복 저장하는 방식은 폐기하고,
연결 테이블로 다대다를 정식으로 풀어냄.

## 왜 "컬럼 많은 넓은 테이블" 대신 여러 테이블+JOIN 구조인가

기관 수, 조사자료 수, 연도 수가 권고마다 다르고 매년 늘어나므로, 이걸 컬럼으로
표현하면(예: inst_1, inst_2... year_2023, year_2024...) 데이터가 늘어날 때마다
테이블 구조 자체를 계속 고쳐야 한다. 행(row)이 늘어나는 구조로 설계하면 테이블
구조 변경 없이 데이터만 계속 추가하면 되므로 이 방식을 택함.
INSERT는 조인이 필요 없고, "참조되는 테이블 먼저, 참조하는 테이블 나중" 순서만
지키면 된다. JOIN은 오직 조회(SELECT) 시점에만 필요.

## 로그인을 "간단한 개인 계정"으로 정한 이유

사용자는 이미 서로 아는 팀원 20명, URL도 비공개. 이런 조건에서는 본격적인
인증 시스템(세션 토큰, 비밀번호 찾기, 이메일 인증 등)이 불필요한 오버엔지니어링이라
판단. AUTH 테이블에 이름+SHA-256(salt) 해시 비밀번호만 두고, 계정은 관리자가
미리 만들어 배포하는 방식으로 단순화.

## FEEDBACK.author_name → auth_id로 변경한 이유

처음엔 "이름을 자유 텍스트로 입력"하는 방식(방법1)을 검토했으나, 최종적으로
AUTH 테이블과 연결하는 방식(방법2)으로 확정. 이유: 팀 공유 비밀번호 하나만으로도
"작성자 신뢰성"을 어느 정도 보장하려면, 로그인한 사람=작성자로 자동 연결되는
구조가 자유 텍스트 입력보다 안전함. 관리 부담(계정 20개 발급)이 크지 않다고 판단.

## 자연어 검색(`/search`) 잠금 — 2026-07-12

원래 `/search`는 text-to-SQL(질문 → SQL 생성 → 실행 → 자연어 답변) 구조였는데,
실사용해보니 팀이 진짜 원하는 자연어 기능은 "몇 개야?/언제야?" 같은 **집계·조회형**이
아니라 "왜 이렇게 됐어?" 같은 **설명·추론형** 질문이었다(예: "2022년 사참위 권고가
있었는데 왜 국회의장은 아무것도 없지?"). 이런 질문에 답하려면 법 조문(사회적참사
진상규명법 등)이나 사참위 종합보고서 원문 같은 근거 문서가 필요한데, 지금 레포엔
`REC.necessity`(권고별 배경 텍스트) 외에 그런 근거 문서가 전혀 없다.

근거 문서 없이 로컬 모델(`llama3.2:1b`)에게 이런 질문을 시켜본 결과, 심지어 단순
SQL 생성조차 존재하지 않는 테이블/컬럼을 지어내는 게 실측으로 확인됐고(예: `"REC"`와
`"INST"`를 없는 컬럼 `rec_no`로 조인, 존재하지 않는 `"RECIPIENT"` 테이블 생성), 하물며
"왜"형 질문에 대한 답은 근거 없이 그럴듯한 서사를 지어낼 위험이 훨씬 크다. 이건 이
프로젝트가 처음부터 제일 경계하던 실패 양상(시스템이 판단·단정해버리는 것)과 정확히
같은 문제라서, 근거 문서를 갖추기 전까지는 기능 자체를 잠그기로 했다.

**결론**: `app/routers/search.py`가 `core.rag.qa`를 호출하지 않고 고정 안내 메시지만
반환하도록 변경(Ollama 호출 경로 자체를 코드에서 제거). Ollama 서비스도
`systemctl stop/disable`로 꺼둠. 재개 조건: 사회적참사진상규명법 조문 + 사참위
종합보고서 원문 같은 근거 문서를 확보해서 이걸 기반으로만 답하게 만들 것 (모델
사전지식에만 의존하는 답변은 채택하지 않음). 이때 모델도 로컬 소형 모델로는 무리라
Claude API 병행이 사실상 필수로 보임("인프라 관련 결정" 항목 참고).

## 보류/미결정 사항

### 권고 간 관계 (연관 권고 링크)
"이 권고와 저 권고가 관련 있다"를 연결하는 자기참조 테이블 아이디어가 나왔으나,
"관련성"의 기준이 모호함(같은 사건 배경/같은 기관/같은 법조항 등 여러 기준 가능).
사람이 수동으로 정할지, LLM 제안+사람 승인으로 반자동화할지, 벡터 유사도로
완전자동화할지 미결정. 실제 데이터를 더 보고 판단하기로 하고 보류.

### 복붙(정체) 탐지 방식 — 구현 완료 (SQL 완전일치 + 벡터 임베딩)
1. **SQL 완전일치**: `IMPL.status` 텍스트가 연도 간 100% 동일한지 (확실한 증거, 🔴)
2. **벡터 임베딩 거리**: 표현은 다르지만 의미상 동일한 재탕까지 탐지 (의심 정황, 🟡)

`app/routers/dashboard.py`에서 두 방식을 배지로 구분 표시 (🔴 "N년째 동일" / 🟡 "실질
변화 없음 의심 (유사도 NN%)"). **모든 연도쌍(n(n-1)/2) 대신 인접 연도만(n-1) 비교하는
것으로 확정** — LAG 윈도우 함수로 "바로 전 연도와" 비교만 하면 되고, 정체 배지는
"지금도 멈춰있는지"가 중요하지 과거 어느 시점의 반복 여부가 아니기 때문.

**임계값**: `scripts/test_embedding_similarity.py`로 실측한 근사-재탕 쌍(괄호 표기만
다름) 코사인 유사도 0.9501, 네거티브 컨트롤(명백히 다른 내용) 0.8462를 기준으로
0.93을 잠정 임계값(`app/stuck.py`의 `SUSPECT_SIMILARITY_THRESHOLD`)으로 잡았다.
실 데이터(275건)에 적용해보니 대부분(유사도 0.99+)은 OCR/PDF 추출 과정의 줄바꿈·
공백 차이뿐인 진짜 재탕이었지만, 임계값 바로 위(0.93~0.94)에서는 실제로 다른
내용인데 우연히 유사도가 높게 나온 경우도 하나 발견됨 — "의심 정황"이지 확정 판정이
아니라고 배지 문구에 명시하고, 원문 status와 유사도 숫자를 그대로 보여줘서 사람이
직접 판단하게 하는 것으로 이 부정확성을 흡수한다. 데이터가 더 쌓이면 임계값 재조정 필요.

### 벡터 저장소: Chroma 대신 Postgres pgvector 확장

project-overview.md/ollama-guide.md는 "벡터 DB(Chroma 등)"이라고 적어뒀었지만, 실제로
필요한 연산을 뜯어보니 "여러 벡터 중 최근접 이웃 찾기"가 아니라 **같은 REC_INST 링크의
인접 연도 status 두 개를 비교하는 1:1 코사인 계산**뿐이었다. 이건 ANN 인덱스가 있는
전용 벡터DB가 풀어야 할 문제가 아니라, 이미 있는 구조화 DB 안에서 컬럼 하나로
충분히 처리 가능한 문제다.

**결론**: Postgres에 `pgvector` 확장을 추가(`docker-compose.yml`을
`pgvector/pgvector:pg16` 이미지로 교체, `core/similarity/schema.sql`로 확장 활성화 +
`IMPL.embedding` 컬럼 추가)하고, `<=>` 코사인 거리 연산자로 SQL 안에서 바로 계산한다.
"의미 기반 탐색(이 권고와 비슷한 다른 권고는?)"처럼 진짜 최근접 이웃 검색이 필요한
기능이 나중에 생기면 그때 Chroma 도입을 다시 검토 — 지금 당장은 별도 서비스를 띄울
이유가 없다고 판단해 보류.

임베딩 모델을 나중에 바꾸면 벡터 공간이 달라져 기존 벡터와 비교가 불가능해지므로,
`IMPL.embedding_model`에 어떤 모델로 만들었는지 같이 저장해서 `core/similarity/
backfill.py`가 모델 불일치 행을 구분해 재생성할 수 있게 했다.

## 기관별 "이행률"을 "이행완료율"이 아니라 "정체 아님 비율"로 정의한 이유

기관별 현황 페이지에 이행률 같은 걸 보여주고 싶었는데, IMPL 테이블에는 "완료 여부"를
나타내는 컬럼이 없고 자유 텍스트 `status`뿐이다. "몇 %가 이행완료됐다"를 구하려면
결국 정부가 스스로 쓴 텍스트를 시스템이 "완료"로 판정해야 하는데, 이건 이 프로젝트가
애초에 의심하는 대상(자기보고 텍스트를 곧이곧대로 믿는 것)과 똑같은 함정이다.
status 텍스트에서 "이행완료" 같은 키워드를 매칭하는 방법도 검토했으나, 같은 문구를
반복하기만 해도 "완료"로 잡혀버려 오히려 정체 은폐를 도와주는 역설이 생길 수 있어 폐기.

**결론**: "이행률" 대신 **"정체 아님 비율"**을 쓴다 — 해당 기관의 REC_INST 링크 중
최근 연도 status가 전년도와 달라진(=`app/stuck.py`의 streak < 2) 링크의 비율.
"완료됐다"가 아니라 "텍스트가 그대로 반복되진 않았다"만 객관적으로 센다. 비교 연도가
2개 미만인 링크는 분모에서 빼고 "자료 부족"으로 따로 표시.
기관 목록 페이지에는 이 비율(%)만, 기관 상세 페이지에는 권고별 행마다 정체/변화 배지
옆에 최신 연도 status 원문을 그대로 노출해서, 실제 완료 여부 판단은 사람이 원문을
읽고 하도록 유도한다.

### 이행현황이 기관별로 다르게 오는지 여부
REC_INST 단위로 IMPL을 붙이도록 설계했지만, 실제로 여러 기관이 걸린 권고에서
연도별 보고가 기관별로 따로 오는지, 통짜로 한 번만 오는지 아직 확인 안 됨.
실제 2023/2024 전체 문서를 더 봐야 확정 가능.

### TERM 테이블에 시맨틱 서치 연동 여부
TERM.definition/context도 벡터DB에 임베딩해서 "OO가 뭐야?" 질문에 활용할지는
아직 옵션 단계. 나중에 실제 구현 시 결정.

## 프론트엔드: React 대신 FastAPI + Jinja2 + HTMX로 시작하는 이유

화면 5개(로그인/대시보드/상세/기관별 현황/용어사전), 사용자 20명 내부 도구인
지금 시점에서 React+별도 API 서버 구조는 빌드 툴체인·상태관리·배포 파이프라인이
추가되는 확실한 비용 대비, "나중에 복잡해질 것"은 아직 가정 단계라 이득이 불분명함.
개발자가 React에 더 익숙하다는 개인적 선호는 있었으나, 백엔드가 이미 Python
스택(Ollama 연동, 파싱 스크립트)으로 굳어져 있어 언어를 통일하는 쪽을 택함.

**결론**: FastAPI가 라우트에서 Jinja2로 HTML을 직접 렌더링(서버사이드 템플릿),
부분 갱신(필터칩, 검색 결과 등)은 HTMX로 처리. FastAPI 자체가 API 프레임워크이므로
나중에 규모가 커지면 같은 라우트를 JSON 응답으로 바꾸거나 `/api` 라우터를 추가하는
식으로 React/Next로 전환하는 경로가 열려 있음 — 지금 구조가 나중 전환을 막지 않음.

## 인프라 관련 결정

- **로컬 LLM(Ollama) 우선, Claude API는 보조**: 데이터가 민감(참사 피해자 관련)하고
  비용도 고려해 기본은 로컬 GPU(RTX 4090)로 처리. 단, 정교한 추론이 필요한 기능
  (예: 검증 항목 자동 분해)은 품질 차이가 크므로 Claude API 사용을 병행 고려.
- **동시 접속 대비**: Ollama 기본 설정은 순차 처리라 20명 동시 접속 시 대기가 발생할
  수 있음. `OLLAMA_NUM_PARALLEL`, `OLLAMA_MAX_QUEUE` 환경변수로 병렬 처리 조정
  가능하나 VRAM 여유(24GB)를 보며 조절 필요.
- **저장소 구조**: DB 스키마, RAG 백엔드, 웹앱 프론트엔드를 하나의 monorepo로
  시작. 팀 규모/배포 방식이 복잡해지면 그때 분리 검토.

## 저장 형식 관련

- `REC.necessity` 등 서술형 텍스트는 마크다운으로 저장. 각주는 블록인용(`>`)으로
  표현해 원문의 부연설명 뉘앙스를 살림.
- `source` 컬럼은 "파일명#page=번호" 형식으로 통일해, 나중에 "원문 보기" 기능에서
  해당 페이지를 바로 열 수 있게 준비.
